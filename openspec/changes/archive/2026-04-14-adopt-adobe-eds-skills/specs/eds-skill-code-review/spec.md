## ADDED Requirements

### Requirement: Skill supports self-review and PR review modes

The skill SHALL operate in two modes: (1) **Self-review** — reviews uncommitted changes in working directory before committing; (2) **PR review** — reviews a specific PR by number or URL.

#### Scenario: Self-review invocation

- **WHEN** invoked without a PR number
- **THEN** the skill SHALL run `git diff` and `git status` to identify changed files and review them against EDS + TypeScript standards

#### Scenario: PR review invocation

- **WHEN** invoked with a PR number
- **THEN** the skill SHALL fetch the PR diff via `gh pr diff <number>`, validate structure (preview URLs, description), and post a review comment with findings

### Requirement: TypeScript-specific review checklist

The skill SHALL validate TypeScript-specific patterns in addition to standard EDS patterns:

- No `any` type usage without justification
- `strictNullChecks` compliance — all `querySelector` results guarded
- Imports use `@/*` alias, not relative paths across modules
- `export default async function decorate(block: HTMLElement): Promise<void>` signature
- `block.replaceChildren()` used instead of `block.innerHTML =` for DOM rebuild
- Unused parameters prefixed with `_`

#### Scenario: Missing null guard

- **WHEN** code has `const el = block.querySelector('.foo'); el.classList.add('bar')` without null check
- **THEN** the skill SHALL flag it as MUST FIX with suggestion to add `if (!el) return;`

### Requirement: Security review enforces DOMPurify for innerHTML

The skill SHALL flag any `innerHTML` assignment that doesn't pass through `DOMPurify.sanitize()` as a BLOCKING security issue.

#### Scenario: Raw innerHTML assignment

- **WHEN** code has `element.innerHTML = someVariable` or `element.innerHTML = fetchedData`
- **THEN** the skill SHALL mark it as **Must Fix (Blocking)** with the suggestion: `element.innerHTML = DOMPurify.sanitize(someVariable)`

### Requirement: CSS review validates scoping and mobile-first breakpoints

The skill SHALL verify: all selectors scoped under `main .{block-name}` or `.{block-name}`; breakpoints use `min-width` (not `max-width`); standard breakpoints are 600px, 900px, 1200px; `!important` not used.

#### Scenario: Unscoped selector

- **WHEN** CSS has `.title { color: red; }` with no block scope
- **THEN** the skill SHALL flag it as Should Fix with suggestion `main .{block-name} .title { color: red; }`

### Requirement: Skill checks that generated output is in src/, not blocks/

The skill SHALL flag any new or modified files in `blocks/`, `scripts/`, or `styles/` (generated directories) as an error — all source changes must be in `src/`.

#### Scenario: File modified in generated directory

- **WHEN** the diff shows a modified file in `blocks/hero/hero.js`
- **THEN** the skill SHALL flag: "Edit `src/blocks/hero/hero.ts` instead — `blocks/` is generated by the build"
